# Dacha alert manager

Сервис подписан на события Frigate через **MQTT (RabbitMQ)**, по тревоге достаёт
медиа из **Frigate API**, складывает в **MinIO** и рассылает авторизованным
пользователям **Telegram**-уведомление с presigned-ссылкой. Управление
уведомлениями и расписанием — из чата.

## Архитектура

```
Frigate ──MQTT(frigate/reviews)──▶ aiomqtt ingress
                                       │  публикует job (persistent, confirmed)
                                       ▼
                         RabbitMQ: dacha.alert.jobs (durable)
                                       │
                                       ▼
                        MediaWorker  ── Frigate API ──▶ MinIO (snap/clip)
                                       │  если should_notify()
                                       ▼
                         RabbitMQ: dacha.alert.outbox (durable)
                                       │
                                       ▼
                        OutboxSender ── через SOCKS-прокси ──▶ Telegram
```

Двухстадийная обработка `frigate/reviews`:

- `type=new`, `severity=alert` → снапшот в MinIO → (если не подавлено) фото-алерт
  всем; подпись `🚨 {camera} · человек · {время}`; сохраняем `review → message_id`.
- `type=end` → клип в MinIO (поллинг с бэкоффом) → к ранее отправленным сообщениям
  цепляем кнопку `🎬 Видео` (presigned-ссылка). Клип — best-effort: если не успел,
  кнопка `🎬 видео недоступно`.

### Почему так переживается смерть прокси

Telegram доступен **только** через прокси. Надёжность построена на **RabbitMQ**,
а не на Postgres:

- медиа скачивается и кладётся в MinIO **до** отправки в TG (это не зависит от прокси);
- единственный шаг, зависящий от прокси, — `OutboxSender`. Если прокси мёртв,
  отправка падает с сетевой ошибкой → задача уезжает в retry-очередь
  (dead-letter + TTL, нарастающий бэкофф) и **остаётся в брокере**;
- ничего не ack-ается до успеха или до передачи в retry-очередь → падение/рестарт
  сервиса не теряет задачу. Как только прокси оживает — уведомления досылаются;
- фан-аут идемпотентен через таблицу `sent_messages` (повторная доставка не
  дублирует сообщения уже получившим их пользователям).

Persistent MQTT-сессия (стабильный `client_id`, `clean_session=false`, QoS 1)
переживает короткие разрывы и не теряет события Frigate.

### Что где хранится

- **Postgres** — только «чистое» состояние: пользователи, синглтон настроек,
  расписание присутствия, идемпотентность медиа (`processed`), карта сообщений
  (`sent_messages`).
- **RabbitMQ** — транзиентные задачи и их ретраи (durable-очереди).
- **MinIO** — снапшоты и клипы (`events/{review_id}/snap.jpg|clip.mp4`).

## Команды бота

| Команда | Кто | Действие |
|---|---|---|
| `/start` | все | регистрация; если не авторизован — заявка суперюзеру |
| `/status` | авториз. | текущий режим и окна на сегодня |
| `/mode` | авториз. | режим алертов (инлайн, 3 состояния) |
| `/schedule` | авториз. | окна присутствия: день → интервалы → ➕/🗑, копирование |
| `/clip` | авториз. | запись последних N сек со всех камер всем авторизованным (видео в чат, фото — fallback) |
| `/users` | суперюзер | список пользователей, выдать/забрать доступ кнопками |
| `/grant <id>` `/revoke <id>` | суперюзер | доступ по ID (работает, даже если человек ещё не писал `/start`) |

Команды показаны в меню бота (`set_my_commands`); у суперюзера меню расширенное.

## Авторизация

Суперюзеры задаются в `.env` (`TG_SUPERUSERS`, список ID через запятую) — они всегда
авторизованы и управляют доступом остальных. Любой, кто нажал `/start`, попадает в
базу как «известный» (но не авторизованный) — суперюзер получает заявку с кнопками
✅/🚫. Можно выдать доступ и по сырому ID (`/grant`), тогда человек станет
авторизован сразу, как только напишет боту. Режим и расписание — глобальные.

## Режим алертов: `should_notify(now, mode, windows)` в TZ дачи

Один глобальный тумблер на три состояния:

- 🔕 **off** — отключены полностью (приехали на дачу, движение ожидаемо) → не слать;
- 🔔 **always** — включены всегда, игнор расписания → слать;
- 📅 **schedule** — по расписанию → слать, кроме окон присутствия на текущий день.

Окна задаются на каждый день недели; интервал через полночь (`22:00-02:00`)
поддержан. Медиа архивируется в MinIO в любом случае — подавляется только отправка.

## Запуск

```bash
cp .env.example .env   # заполнить токены/хосты/прокси
docker compose up -d --build
```

В compose поднимаются **бот + Postgres**. RabbitMQ, MinIO и Frigate — внешние,
бот ходит к ним исходящими соединениями; Telegram — через `TG_PROXY_URL`.

Локально без Docker:

```bash
uv sync
uv run python -m telegram_alert
```

## Предварительные шаги вне сервиса

- В Frigate включить публикацию `mqtt` на ваш RabbitMQ (host/port/user/pass/vhost);
  зоны/алерты уже настроены.
- Завести в Frigate отдельного пользователя под бота (`FRIGATE_USER/PASSWORD`).
- На RabbitMQ должен быть включён MQTT-плагин (для ingress) и доступен AMQP
  (5672) для внутренних очередей.

## Конфигурация

Все переменные сгруппированы по префиксам в [.env.example](.env.example):
`APP_`, `TG_`, `DB_`, `MQTT_`, `AMQP_`, `FRIGATE_`, `MINIO_`. Реализация —
`pydantic-settings`, по классу на группу ([telegram_alert/config.py](telegram_alert/config.py)).

## Остаточный риск

При server-side схеме, если питание дачи режут сразу после детекта, снапшот может
не успеть выкачаться (медиа лежит на дачном SSD). Это осознанная плата за отказ от
дачного аплоадера.
