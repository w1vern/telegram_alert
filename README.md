# Dacha alert manager

Сервис подписан на события Frigate через **MQTT (RabbitMQ)**, по тревоге достаёт
медиа из **Frigate API**, складывает в **MinIO** и рассылает авторизованным
пользователям **Telegram**-уведомление с presigned-ссылкой. Управление
уведомлениями и расписанием — из чата.

## Архитектура

```
Frigate ──MQTT(frigate/reviews)──▶ aiomqtt ingress
                                       │  публикует job (persistent, confirmed)
                                       ▼
                         RabbitMQ: dacha.alert.jobs (durable)
                                       │
                                       ▼
                        MediaWorker  ── Frigate API ──▶ MinIO (snap/clip)
                                       │  если should_notify()
                                       ▼
                         RabbitMQ: dacha.alert.outbox (durable)
                                       │
                                       ▼
                        OutboxSender ── через SOCKS-прокси ──▶ Telegram
```

Двухстадийная обработка `frigate/reviews`:

- `type=new`, `severity=alert` → снапшот в MinIO → (если не подавлено) фото-алерт
  всем; подпись `🚨 {camera} · человек · {время}`; сохраняем `review → message_id`.
- `type=end` → клип в MinIO (поллинг с бэкоффом) → к ранее отправленным сообщениям
  цепляем кнопку `🎬 Видео` (presigned-ссылка). Клип — best-effort: если не успел,
  кнопка `🎬 видео недоступно`.

### Почему так переживается смерть прокси

Telegram доступен **только** через прокси. Надёжность построена на **RabbitMQ**,
а не на Postgres:

- медиа скачивается и кладётся в MinIO **до** отправки в TG (это не зависит от прокси);
- единственный шаг, зависящий от прокси, — `OutboxSender`. Если прокси мёртв,
  отправка падает с сетевой ошибкой → задача уезжает в retry-очередь
  (dead-letter + TTL, нарастающий бэкофф) и **остаётся в брокере**;
- ничего не ack-ается до успеха или до передачи в retry-очередь → падение/рестарт
  сервиса не теряет задачу. Как только прокси оживает — уведомления досылаются;
- фан-аут идемпотентен через таблицу `sent_messages` (повторная доставка не
  дублирует сообщения уже получившим их пользователям).

Persistent MQTT-сессия (стабильный `client_id`, `clean_session=false`, QoS 1)
переживает короткие разрывы и не теряет события Frigate.

### Что где хранится

- **Postgres** — только «чистое» состояние: пользователи, синглтон настроек,
  расписание присутствия, идемпотентность медиа (`processed`), карта сообщений
  (`sent_messages`).
- **RabbitMQ** — транзиентные задачи и их ретраи (durable-очереди).
- **MinIO** — снапшоты и клипы (`events/{review_id}/snap.jpg|clip.mp4`).

## Команды бота

| Команда | Кто | Действие |
|---|---|---|
| `/start` | все | регистрация; если не авторизован — заявка суперюзеру |
| `/status` | авториз. | текущий режим, активный override и окна на сегодня |
| `/mode` | авториз. | режим алертов (инлайн, 3 состояния); смена режима сбрасывает override |
| `/schedule` | авториз. | окна присутствия: день → интервалы → ➕/🗑, копирование |
| `/clip <сек>` | авториз. | запись последних N сек со всех камер всем авторизованным (фото + кнопка на видео); длина обязательна |
| `/mute <часы>` `/unmute <часы>` `/auto` | авториз. | временный override поверх расписания на N часов (дробь ок): заглушить / включить / сбросить (только в режиме «по расписанию») |
| `/users` | суперюзер | список пользователей, выдать/забрать доступ кнопками |
| `/grant <id>` `/revoke <id>` | суперюзер | доступ по ID (работает, даже если человек ещё не писал `/start`) |

Меню бота (`set_my_commands`) подстраивается под права в реальном времени, по
`BotCommandScopeChat`. Три уровня: **неавторизованный** (`/start`, `/help`),
**авторизованный** (полный набор управления) и **суперюзер** (плюс `/users`,
`/grant`, `/revoke`). Меню обновляется на `/start`, при выдаче и отзыве доступа,
а при старте сервиса — для всех уже известных авторизованных и суперюзеров.
Неавторизованный вызов команды управления получает понятный отказ.

## Авторизация

Суперюзеры задаются в `.env` (`TG_SUPERUSERS`, список ID через запятую) — они всегда
авторизованы и управляют доступом остальных. Любой, кто нажал `/start`, попадает в
базу как «известный» (но не авторизованный) — суперюзер получает заявку с кнопками
✅/🚫. Можно выдать доступ и по сырому ID (`/grant`), тогда человек станет
авторизован сразу, как только напишет боту. Режим и расписание — глобальные.

## Режим алертов: `should_notify(now, mode, windows)` во времени дачи

Время дачи задаётся числом — смещение от UTC в часах (`APP_UTC_OFFSET`, напр.
Москва `3`, Магадан `11`). Это единственный пояс в системе: метки событий Frigate
абсолютны (UTC), время сервера не используется. Смещение нужно лишь чтобы
трактовать окна расписания и форматировать время в сообщениях по местному времени
камер.


Один глобальный тумблер на три состояния:

- 🔕 **off** — отключены полностью (приехали на дачу, движение ожидаемо) → не слать;
- 🔔 **always** — включены всегда, игнор расписания → слать;
- 📅 **schedule** — по расписанию → слать, кроме окон присутствия на текущий день.

Окна задаются на каждый день недели; интервал через полночь (`22:00-02:00`)
поддержан. Медиа архивируется в MinIO в любом случае — подавляется только отправка.

### Временный override (только режим «по расписанию»)

Поверх расписания есть один глобальный временный тумблер с обязательным сроком
(`settings.override` + `settings.override_until`, epoch):

- `/mute <часы>` — заглушить на N часов, игнорируя окна (приехали вне окна);
- `/unmute <часы>` — включить на N часов, игнорируя окна (вышли, ждём гостей);
- `/auto` — сбросить override сразу, вернуться к чистому расписанию.

Срок обязателен (без аргумента команда не выполняется), допустима дробь
(`/mute 0.5`), потолок — 7 дней. По истечении срока override сам считается
снятым (ленивая проверка в `resolve_override`, отдельной чистки БД нет).
Новый `/mute`/`/unmute` заменяет предыдущий и переустанавливает таймер; смена
режима через `/mode` тоже сбрасывает override. В режимах «отключены»/«включены
всегда» команды не действуют. Активный override и остаток времени видны в
`/status`.

### Единый вид сообщений

Алерт по событию и `/clip` присылаются одинаково: **фото с подписью + кнопка
`🎬 Видео`** на presigned-ссылку MinIO. Отличается только подпись
(`🚨 … человек` vs `🎥 … запись`). У события кнопка появляется позже (когда клип
дозапишется, по `type=end`); у `/clip` фото и клип готовы сразу, поэтому кнопка
сразу на месте. Если клипа нет — приходит только фото.

## Запуск

```bash
cp .env.example .env   # заполнить токены/хосты/прокси
docker compose up -d --build
```

В compose поднимаются **бот + Postgres + миграции**. RabbitMQ, MinIO и Frigate —
внешние, бот ходит к ним исходящими соединениями; Telegram — через `TG_PROXY_URL`.

Локально без Docker (схему предварительно накатить через `make migration`):

```bash
uv sync
uv run python -m telegram_alert
```

### Миграции (Alembic)

Схема БД управляется **Alembic**. Миграции вынесены в отдельный one-shot сервис
`migrate` в [compose.yml](compose.yml): он прогоняет `alembic upgrade head` и
завершается, а `bot` стартует только после его успешного завершения
(`depends_on … service_completed_successfully`). Сам бот схему не трогает —
только досоздаёт singleton-настройку. DSN берётся из настроек приложения
(`.env`), отдельной строки подключения в `alembic.ini` нет.

Конфигурация и ревизии лежат в пакете: [telegram_alert/migrations/](telegram_alert/migrations/).

Команды — в [Makefile](Makefile), все идут через docker compose, поэтому Postgres
поднимается автоматически и хост `postgres` резолвится внутри сети compose
(отдельно поднимать БД и подменять `DB_HOST` не нужно):

```bash
make migration                    # alembic upgrade head (накатить всё)
make gen_migration m="add field"  # автогенерация миграции по моделям
make down_migration               # откатить одну миграцию
```

`make gen_migration` пишет новый файл в
[telegram_alert/migrations/versions/](telegram_alert/migrations/versions/)
(каталог примонтирован в сервис `migrate`), так что ревизия сразу появляется на
хосте.

## Предварительные шаги вне сервиса

- В Frigate включить публикацию `mqtt` на ваш RabbitMQ (host/port/user/pass/vhost);
  зоны/алерты уже настроены.
- Завести в Frigate отдельного пользователя под бота (`FRIGATE_USER/PASSWORD`).
- На RabbitMQ должен быть включён MQTT-плагин (для ingress) и доступен AMQP
  (5672) для внутренних очередей.

## Конфигурация

Все переменные сгруппированы по префиксам в [.env.example](.env.example):
`APP_`, `TG_`, `DB_`, `MQTT_`, `AMQP_`, `FRIGATE_`, `MINIO_`. Реализация —
`pydantic-settings`, по классу на группу ([telegram_alert/config.py](telegram_alert/config.py)).

## Остаточный риск

При server-side схеме, если питание дачи режут сразу после детекта, снапшот может
не успеть выкачаться (медиа лежит на дачном SSD). Это осознанная плата за отказ от
дачного аплоадера.
